# 贡献指南

即使是个人项目，也建议按固定约定提交与合并，确保随时可回退、可定位问题。

## 分支约定

- `main`：稳定分支，保证可构建/可运行
- `feature/*`：新增功能
- `fix/*`：修复问题
- `chore/*`：杂项（依赖升级、构建配置、重构等）
- `experiment/*`：实验分支（随时可以丢弃）

## 提交信息建议

推荐使用简洁的前缀，便于检索：

- `feat:` 新功能
- `fix:` 修复
- `refactor:` 重构
- `chore:` 构建/工程化
- `docs:` 文档

示例：

- `feat: add product price history chart`
- `fix: handle missing record id`

## 提交粒度

- 每次提交尽量保持“能跑起来”（能 `npm run build` 或至少不破坏开发运行）
- 避免把不相关改动混在同一次提交里

## 不要提交的内容

- 任何密钥与个人环境配置（例如 `.env*`）
- 构建产物（例如 `dist/`）

## 合并策略（个人开发推荐）

- 功能完成后再合入 `main`
- 功能效果不理想：直接删除分支，不要在 `main` 上反复试错
- 已经推到远端且可能被其他环境引用的改动：优先用 `git revert` 撤销，避免改写历史
